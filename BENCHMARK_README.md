# Сравнение MPC: бенчмарк и метрики

Этот документ описывает, **как устроен проект `mpc_comparing` с точки зрения сравнения решателей**, как запускается скрипт **`benchmark_compare.py`**, что означает **каждая метрика** и **каждый выходной график**, и какие **ограничения** у интерпретации результатов.

---

## 1. Что именно сравнивается

В репозитории три замкнутых контура (closed loop) в **ir-sim**: один и тот же сценарий (YAML), одна и та же идея планирования (прямой глобальный путь от старта к цели, локальное окно reference, receding horizon), но **разный численный бэкенд** внутри MPC.

| Метод | Модуль MPC | Симуляция-обёртка |
|--------|------------|-------------------|
| **IPOPT** | `MPC/unicycle_mpc_ipopt_tracking.py` — нелинейная постановка (NLP), ограничения на препятствия как точное условие на расстояние | `simulation/UnicycleMPC_Tracking.py` |
| **OSQP** | `MPC/unicycle_mpc_osqp_tracking.py` — линеаризованный QP, линейное приближение препятствий, поверх решения — **safety filter** | `simulation/UnicycleMPC_Tracking_OSQP.py` |
| **Active Set (qpOASES)** | `MPC/unicycle_mpc_active_set_tracking.py` — та же QP-постановка, что у OSQP, другой QP-решатель | `simulation/UnicycleMPC_Tracking_ActiveSet.py` |

**Смысл сравнения в терминах эффективности:**

- **IPOPT** обычно точнее отражает нелинейную динамику и геометрию препятствий, но решение NLP дороже по времени.
- **OSQP** и **Active Set** решают приближённую QP-задачу быстрее, но аппроксимация ограничений и линеаризация могут ухудшать поведение в сложной геометрии; safety filter снижает риск немедленного столкновения на следующем шаге.

Подробная математика и различия постановок описаны в основном **`README.md`** проекта.

## 2. Одинаковые настройки и что всё ещё отличается

**Общие** параметры трекинга (горизонт, **`q_pos` / `q_theta` / `r` / `r_yaw`**, **`integration_method`**, лимиты **`v_max` / `w_max` / `dv_max` / `dw_max`**, **`safe_distance`**, **`n_obs`**) заданы **одинаково** для IPOPT, OSQP и Active Set в **`benchmark_compare.py`** и в **`simulation/UnicycleMPC_Tracking*.py`**.

У OSQP и Active Set дополнительно совпадают параметры QP-оболочки (`slack_weight`, `obs_horizon`, `activation_margin` и остальные поля из общего блока в двух скриптах).

**Математика разная**: IPOPT — NLP с нелинейными ограничениями по препятствиям; QP-ветки — линеаризация, линейные полуплоскости и safety filter. Сравниваются постановки при согласованных весах, а не одна и та же задача в трёх копиях.

### Типичная картина по `metrics.jsonl` (полный прогон всех обнаруженных сценариев)

Ориентиры по **средним** по сценариям (конкретные цифры зависят от машины и версии кода; пересчитайте после своего прогона):

| Солвер | `solve_mean_ms` | `solve_p95_ms` | `cte_rmse` | `steps` |
|--------|-----------------|----------------|------------|---------|
| IPOPT | выше (~14 ms) | выше | **ниже** (~0.10) | чуть меньше |
| OSQP | средне (~9–10 ms) | **ниже хвост** (~12 ms) | выше (~0.15) | чуть больше |
| Active Set | **часто минимум среднего** (~9 ms) | хвост иногда выше OSQP | как у OSQP (~0.15) | как у OSQP |

На последнем полном прогоне (число сценариев = длина списка из `_discover_scenarios()`) у всех трёх часто **успех по всем картам** по критерию `final_dist < 0.2` — пересчитайте по своему `metrics.jsonl`. Смысл графиков в `benchmark_results/`: не только средняя задержка, но и **распределение** времён (ECDF, max) и компромисс время ↔ точность слежения за прямой (`scatter_time_vs_cte.png`). Подробный инженерный разбор методов и история отладки — в **`MPC/explanation.md`**.

## 3. Как проходит бенчмарк (пошагово)

1. **Рабочая директория** — корень репозитория `mpc_comparing` (чтобы импорты `MPC.*` и пути к `scenarios/` были корректны).

2. Скрипт **`benchmark_compare.py`** выставляет **неинтерактивный бэкенд Matplotlib** (`Agg`), чтобы графики сохранялись без дисплея.

3. **Обнаружение сценариев**: для каждой подпапки `scenarios/<имя>/` проверяется наличие файла `scenarios/<имя>/<имя>.yaml`. Список сценариев отсортирован по имени папки.

4. Для **каждого сценария**:
   - один раз создаётся среда **ir-sim** по YAML, читается **цель** `goal`, среда закрывается;
   - для **каждого** из трёх солверов **последовательно** вызывается `run_one_solver`:
     - снова создаётся среда по тому же YAML;
     - строится объект MPC через соответствующую фабрику (`make_*_mpc`), в неё передаётся `env.step_time` и остальные параметры;
     - вызывается **`run_simulation(..., verbose=False, render=False)`** из соответствующего файла в `simulation/`:
       - **без OpenCV** (нет `imshow`), удобно для серверов и CI;
       - без лишнего вывода в консоль на каждом шаге;
     - после прогона среда закрывается;
     - из сырого словаря результата и цели строится расширенный словарь **метрик** (`enrich_metrics`).

5. Если для сценария удалось получить хотя бы один успешный прогон, строится **дашборд** `dashboard_<сценарий>.png` (см. раздел про графики).

6. После всех сценариев:
   - в **`metrics.jsonl`** пишется по одной JSON-строке на каждый успешный прогон (солвер × сценарий);
   - строятся **агрегированные** графики по всем накопленным метрикам.

7. Ошибка при запуске одного солвера (например, отсутствие **qpOASES** в сборке CasADi для Active Set) **не останавливает** остальные: в консоли будет строка `ОШИБКА`, этот солвер пропускается для данного сценария.

## 4. Порог успеха и что попадает в `metrics.jsonl`

- **`success`**: `true`, если **`final_dist < 0.2`**, где `final_dist` — евклидово расстояние от последней позиции робота до цели в плоскости XY **после последнего применённого управления**. Этот же порог **0.2** используется в симуляциях как условие «цель достигнута».

- В **`metrics.jsonl`** попадают **все скалярные поля** метрик, **кроме** массива **`solve_times_ms`** (он нужен только для построения гистограммы в памяти и сильно раздувает файл).

## 5. Словарь метрик (что означает каждое поле)

Ниже — поля, которые формирует **`enrich_metrics`** в `benchmark_compare.py`. Имена совпадают с ключами в `metrics.jsonl` (кроме исключения выше).

### Идентификация прогона

| Поле | Смысл |
|------|--------|
| `solver` | Строка: `IPOPT`, `OSQP` или `ActiveSet`. |
| `integrator` | Дискретизация динамики в постановке MPC: `euler` или `rk4` (задаётся флагом `--integrators`). |
| `run_label` | Уникальная метка серии на графиках: `{solver}_{integrator}` (например `OSQP_rk4`). |
| `scenario` | Имя папки сценария (например, `free_path`, `head_on`). |

### Исход симуляции

| Поле | Смысл |
|------|--------|
| `steps` | Число **итераций MPC**, равное длине списка `solve_times` (по одному решению на шаг после каждого `env.step`). |
| `final_dist` | Расстояние до цели (XY) в конце прогона. |
| `success` | `final_dist < 0.2`. |
| `dt` | Шаг времени среды `env.step_time` из YAML (секунды). |
| `wall_time_sim` | **`steps * dt`** — модельное «физическое» время эпизода по часам симулятора (не CPU-время). |

### Геометрия пути

| Поле | Смысл |
|------|--------|
| `path_length` | Длина пройденной траектории в XY: сумма длин отрезков между последовательными точками `trajectory[:, :2]`. |
| `straight_dist` | Расстояние по прямой от **первой** точки траектории до **цели** (XY). |
| `path_efficiency` | **`straight_dist / path_length`**. Показывает, насколько фактический путь близок к «идеальной» прямой от старта к цели по длине: значение ближе к **1** означает меньше лишнего объезда; при сильном обходе препятствий путь длиннее, эффективность ниже. |

### Поперечная ошибка (CTE) относительно прямого reference

Опорная линия — **`reference_path`** из результата симуляции: та же **прямая** от старта к цели, по которой строится локальное окно для MPC.

Для каждой точки траектории считается **минимальное расстояние** от этой точки до **ломаной** из отрезков `reference_path` (проекция на отрезок, параметр обрезан в \([0,1]\)).

| Поле | Смысл |
|------|--------|
| `cte_rmse` | \(\sqrt{\mathrm{mean}(\mathrm{CTE}^2)}\) по всем точкам траектории — **среднеквадратичная** поперечная ошибка относительно прямой. |
| `cte_max` | Максимум модуля поперечной ошибки по эпизоду. |

Если по какой-то причине нет валидного `reference_path`, обе величины становятся `NaN`.

### Время решения MPC (латентность решателя)

Из `result["solve_times"]` (секунды на один вызов `solve`):

| Поле | Смысл |
|------|--------|
| `solve_mean_ms` | Среднее время решения за шаг, в миллисекундах. |
| `solve_median_ms` | Медиана, мс. |
| `solve_p95_ms` | 95-й перцентиль, мс — типичный показатель «хвоста» задержек для soft real-time. |
| `solve_p99_ms` | 99-й перцентиль, мс. |
| `solve_max_ms` | Максимум за эпизод, мс. |
| `total_solve_s` | Сумма всех `solve_times` за эпизод, секунды — **полное CPU-время** на решение NLP/QP по всем шагам (не параллелится между шагами в одном прогоне). |

Массив **`solve_times_ms`** используется **только внутри** `benchmark_compare.py` для гистограммы и **не пишется** в `metrics.jsonl`.

### Управление и гладкость

| Поле | Смысл |
|------|--------|
| `control_energy` | \(\sum_k (v_k^2 + \omega_k^2)\) по применённым управлениям — **прокси «энергии»** управления (без весов как в функционале MPC). |
| `jerk_proxy_v` | \(\sum_k (v_{k+1} - v_k)^2\) — **прокси рывка** по линейной скорости. |
| `jerk_proxy_w` | \(\sum_k (\omega_{k+1} - \omega_k)^2\) — **прокси рывка** по угловой скорости. |

Большие значения обычно соответствуют более «дёрганому» управлению; интерпретация зависит от длины эпизода и числа шагов.

## 6. Выходные файлы и смысл графиков

По умолчанию каталог: **`benchmark_results/`** (задаётся флагом `--output`).

### 6.1. Дашборд по сценарию: `dashboard_<имя>.png`

Одна большая фигура на сценарий, на которой **для всех успешно прогнанных солверов** на этом YAML:

- **Траектории XY** — кто сильнее отклоняется от прямой, кто шире обходит препятствия.
- **Текстовая сводка** — `success`, `steps`, среднее и p95 времени решения, `cte_rmse`.
- **Дистанция до цели по шагам** — скорость сходимости; горизонтальная линия на **0.2** — порог успеха.
- **Время решения по шагам (мс)** — задержка MPC в каждой итерации.
- **Накопленное время решателя (с)** — интеграл задержек; полезно сравнивать суммарную вычислительную нагрузку за эпизод.
- **Поперечная ошибка (CTE) по шагам** — насколько робот «режет углы» или уходит в сторону от прямой reference.
- **Норма управления \(\sqrt{v^2+\omega^2}\)** — насколько агрессивны команды.
- **ECDF времени решения** — эмпирическая функция распределения задержек: видно, какой доле шагов соответствует, например, время ниже заданного порога.

### 6.2. Агрегированные графики (все сценарии)

| Файл | Смысл |
|------|--------|
| `aggregate_timing_bars.png` | Две панели: **среднее** и **p95** времени решения (мс) **по каждому сценарию** и каждому солверу — удобно сравнивать типичную и «хвостовую» латентность. |
| `aggregate_steps.png` | **Число шагов** симулятора до остановки (успех по расстоянию, `env.done()` или лимит `--max-steps`). Отражает, сколько итераций потребовалось задаче. |
| `heatmap_success.png` | Матрица **сценарий × солвер**: достигнута ли цель (`success`). Визуально показывает **надёжность** по сетке тестов. |
| `bar_cte_rmse_by_scenario.png` | **RMSE поперечной ошибки** по сценариям — **качество слежения** относительно прямой линии reference. |
| `scatter_time_vs_cte.png` | Каждая точка — один прогон: **среднее время решения** vs **CTE RMSE**. Показывает **компромисс** «быстрее считать» vs «ближе держаться прямой». |
| `hist_solve_times_pooled.png` | **Нормированные гистограммы** всех покадровых времён решения, объединённых по всем сценариям, по солверам — форма распределения латентности. |
| `box_mean_solve_time.png` | **Box plot** распределения **среднего** времени решения по сценариям для каждого солвера — разброс «типичной» скорости между картами. |
| `radar_normalized.png` | **Полярный график**: по осям — нормализованные показатели (обратные к среднему времени и к p95, обратное к среднему CTE, средняя `path_efficiency`, доля успешных сценариев). Для каждого солвера строится **профиль** «лучше по всем осям — ближе к внешнему контуру» внутри одной нормализации по максимуму среди солверов. |

### 6.3. Машиночитаемый отчёт

| Файл | Смысл |
|------|--------|
| `metrics.jsonl` | Одна строка JSON на прогон; удобно для скриптов, pandas, постобработки. Поле `solve_times_ms` **намеренно отсутствует**. |

## 7. Запуск из командной строки

Из корня репозитория, с активированным виртуальным окружением (если используется):

```bash
cd /path/to/mpc_comparing
source .venv/bin/activate   # при необходимости
python benchmark_compare.py
```

Полезные аргументы:

| Аргумент | Описание |
|----------|-----------|
| `--scenario <имя>` | Только один сценарий (имя папки в `scenarios/`, например `head_on`). |
| `--output <каталог>` | Куда сохранять PNG и `metrics.jsonl` (по умолчанию `benchmark_results`). |
| `--max-steps N` | Верхняя граница шагов в `run_simulation` (по умолчанию 500). |
| `--integrators A,B` | Интеграторы для **каждого** солвера: `euler`, `rk4` или оба через запятую (`euler,rk4`). По умолчанию только `euler`. Полный прогон с двумя методами даёт **6** закрытых контуров на сценарий (3 солвера × 2 интегратора). |

Примеры:

```bash
python benchmark_compare.py --scenario free_path
python benchmark_compare.py --output ./my_run --max-steps 300
python benchmark_compare.py --integrators euler,rk4
```

Переменная **`MPLBACKEND=Agg`** обычно не обязательна: скрипт сам переключает Matplotlib на Agg в начале файла.

## 8. Связь с интерактивными симуляциями

Файлы `simulation/UnicycleMPC_Tracking*.py` по умолчанию запускают **визуализацию OpenCV** и печать в консоль. Для бенчмарка используется тот же `run_simulation`, но с **`render=False`** и **`verbose=False`**, чтобы:

- не требовался дисплей и окна;
- не замедлять прогон отрисовкой;
- уменьшить шум в логах.

Поведение контроллера и среды при этом **то же самое**, что при интерактивном запуске с теми же `mpc_config`.

---

## 9. Краткая шпаргалка по интерпретации

- **Низкое среднее / p95 времени решения** — лучше для реалтайм-потока с жёстким периодом дискретизации.
- **Низкий `cte_rmse`** — плотнее следование **прямой** от старта к цели (не путать с глобально оптимальным обходом: прямая в проекте намеренно примитивна).
- **Высокий `path_efficiency`** — меньше лишней длины пути относительно прямой «как вороной»; при плотных препятствиях низкая эффективность может быть **нормальной ценой** безопасного объезда.
- **`success` на heatmap** — главный индикатор «задача доехала до цели в рамках порога» для данной карты и настроек.
- **Радар и scatter** — агрегированные «портреты» компромиссов; их стоит читать вместе с **отдельными дашбордами** по сложным сценариям.

Если нужна ревизия документа под вашу ветку (другие пороги, унифицированные веса, дополнительные метрики), имеет смысл обновить этот файл и комментарии в `benchmark_compare.py` синхронно.
